# 忆路康（YiLuKang）

> 面向银龄群体的智能认知健康管理平台

## 项目简介

忆路康是一个面向银龄群体的智能认知健康管理平台，覆盖**认知筛查评估、医疗知识咨询、个性化干预、情感陪伴、多角色协同管理**五大核心场景。系统基于知识图谱与大语言模型，通过 LangGraph Agent 架构实现智能问答，集成三路 RAG（图谱/向量/关键词）+ RRF 融合检索。同时，系统支持行为分析、NASA-TLX 问卷与 EEG 生理信号三模态融合的认知负荷量化评估，为认知障碍的早期发现与长期管理提供数据支撑。

### 核心能力

- **AI Agent 知识引擎**：基于 LangGraph 的 Agent 架构，LLM 自主调用 12+ 工具（图检索、向量检索、混合检索、患者教育、科普推文等），支持多轮对话与会话记忆
- **三路 RAG + RRF 融合**：图谱检索（Neo4j）+ 向量检索（ChromaDB）+ 关键词检索，通过 Reciprocal Rank Fusion 统一排序去重
- **多模态认知负荷评估**：行为埋点 + NASA-TLX 主观问卷 + EEG 生理信号（DEAP 公开数据集仿真 / Muse S 硬件预留），多模态加权融合量化认知负荷
- **多角色协作管理**：机构管理员、医生、家属、老人四角色闭环，各角色独立视图与权限控制
- **适老化交互设计**：WCAG ~95% 合规，大字号/高对比/语音输入/TTS 朗读/简化导航，尊重式交互

## 功能模块

### 一、智能咨询

| 功能 | 说明 |
|------|------|
| Agent 智能问答 | LangGraph 编排 LLM + RAG 工具链，支持多轮对话、流式输出、来源溯源 |
| 患者教育 | 一键生成结构化患者教育内容（标题 + 分节 + 温馨提示），支持长文/分步/卡片多种呈现 |
| 科普推文 | 将问答结果概括为 1-3 条科普推文，附推荐话题标签 |
| 深度思考 | 展示 Agent 检索与推理过程，增强可信度 |
| 多模型切换 | 支持 DeepSeek / GPT / Gemini / 通义千问 / 豆包 / 智谱 / 文心一言 / Kimi / Grok |
| 语音输入 | 集成 Whisper 本地模型，支持语音转文字 |
| 图片解读 | OCR + LLM 通俗解释，作为 Agent 工具与聊天入口 |

### 二、认知筛查评估

| 功能 | 说明 |
|------|------|
| NASA-TLX 问卷 | 简化版六维度认知负荷量表（1-7 分滑动评分），适老化设计 |
| 行为分析 | 基于点击/回退/错误/耗时等行为特征的实时认知负荷评估 |
| EEG 信号采集 | Muse S 模拟器已实现；硬件接口预留，实验阶段采用 DEAP 公开数据集仿真 |
| 多模态融合评分 | 问卷 60% + 行为 40%（+ EEG 可选），权重自适应 |
| 评估报告 | 单次报告 + 趋势分析 + 周期汇总，雷达图、波动分析、风险预警、个性化建议 |
| 实验测试流程 | 5 步完整流程——信息注册 → 基线问卷 → 系统交互（行为埋点）→ 后测问卷 → 评分报告 |

### 三、个性化干预

| 功能 | 说明 |
|------|------|
| 服药提醒 | 用药清单 + 今日任务 + 进度反馈，支持认知负荷埋点与问卷触发 |
| 患者教育生成 | 智能生成个性化教育内容，支持自适应推荐模式 |
| 首页日程 | 三件小事 + 今日日程（完成/提醒中/错过/推迟），时间动态调整 |

### 四、情感陪伴（小忆）

| 功能 | 说明 |
|------|------|
| 尊重式对话 | 认知障碍 ≠ 智力障碍，尊重老人人生阅历，去除哄小孩语气 |
| 记忆机制 | 从对话中提取关键信息，让小忆"认识"老人 |
| 健康常识问答 | 区别于通用聊天机器人的核心差异化能力 |
| 多主题切换 | 科技风 / 医疗风 / 适老模式，持久化 |
| TTS 朗读 | 智能选音（优先微软晓晓），支持全文朗读 |
| 情绪指示器 | 动态计算对话情绪状态 |

### 五、协同管理

| 功能 | 说明 |
|------|------|
| 机构看板 | 管理员/医生跨用户聚合统计（事件、问卷、活跃趋势、来源分布），支持 CSV 导出 |
| 家属周报 | 本周服药/认知状态汇总、活动记录与照护建议 |
| 角色权限控制 | 四角色 RBAC，各角色独立视图与功能入口 |
| 认知档案 | 个体化认知状态记录，知识图谱构建用户认知画像 |

### 六、知识图谱与数据管理

| 功能 | 说明 |
|------|------|
| 知识图谱生成 | PDF（正文+表格+OCR）/ 文本 / JSON 多数据源，LLM 实体关系提取 + 后处理校验 |
| 多图谱管理 | 图谱切换、实体搜索与筛选、节点详情、导出 JSON/图片 |
| 文档知识库 | 个人/共享知识库、文献检索、多知识库管理 |
| Hadoop 分布式处理 | 大规模数据场景可选组件（云服务器未启用，本地可用） |
| 批量构建 | 本地多线程并行构建，可选 Hadoop MapReduce 预处理 |
| 异步任务 | Celery 任务队列，支持批量图谱构建、HDFS 上传等 |

### Docker 一键部署（4 步）

```bash
# 1. 克隆项目
git clone https://github.com/xiangyun247/knowledge_graph.git
cd knowledge_graph

# 2. 配置环境变量
cp .env.example .env
# 编辑 .env 填写必要参数（见下方说明）

# 3. 启动所有服务
docker compose up -d

# 4. 验证
docker compose ps
curl http://localhost:5001/health
```

> 注：Hadoop 相关容器需 8GB+ 内存，资源有限时可仅启动核心服务：`docker compose up -d neo4j mysql redis backend celery-worker celery-beat`

### 本地开发模式

**后端：**

```bash
pip install -r requirements.txt
docker compose up -d neo4j mysql redis   # 仅启动数据库
python run.py                             # 启动后端，默认监听 5001
```

```bash
# 在前端仓库根目录

# 配置 .env.development
```

## 典型使用流程

### 知识图谱问答

1. 上传 PDF/文本到知识库，系统自动构建知识图谱
2. 在问答框输入医学问题，Agent 自动检索图谱和文献并生成回答
3. 查看回答来源（图谱实体 / 文献片段），支持多轮追问

### 认知负荷评估

1. 填写受试者信息，选择基线参照（如豆包/微信）
2. 完成前测问卷（NASA-TLX 六维度）→ 与小忆对话（行为埋点采集）→ 完成后测问卷
3. 查看评估报告：多模态融合评分、雷达图、趋势分析、个性化建议

### 多角色协同

1. **机构管理员**：查看机构看板，跨用户聚合统计与数据导出
2. **医生**：查看患者认知档案，评估报告，个性化干预建议
3. **家属**：查看家属周报，了解老人认知状态与照护建议
4. **老人**：适老简化模式，三入口导航，服药提醒，与小忆聊天

## Hadoop 分布式处理

基于 Hadoop 3.3.0+ 的分布式文本处理模块，适用于大规模数据场景。**当前云服务器未启用，本地开发环境可用。**

### 架构

- **MapReduce 管线**：
  - 文本提取（PDF → 文本）：`hadoop/mapreduce/pdf_extract/`
  - 文本清理：`hadoop/mapreduce/text_clean/mapper.py`
  - 文本分块：`hadoop/mapreduce/text_chunk/mapper.py`

### 使用策略

- 批量构建默认使用本地多线程并行，不依赖 Hadoop
- 仅当 `use_hadoop=true` 且文件缺少本地提取文本时，才对部分文件走 MapReduce
- 少量文件时 MapReduce 启动开销大于本地多线程，适合文件量较大时启用

## 常见问题

| 问题 | 解决方案 |
|------|---------|
| `docker compose up` 报端口占用 | 修改 `.env` 中对应端口，或停止占用进程 |
| 后端 healthcheck 显示 unhealthy | 检查 `.env` 配置是否完整，特别是 `DEEPSEEK_API_KEY` |
| Embedding 模型首次加载慢 | CPU 正常现象，或设 `USE_LOCAL_EMBEDDING=False` |
| `docker compose restart` 后配置未生效 | 使用 `docker compose up -d` 重新创建容器 |
| Hadoop 容器启动失败 | 内存不足，最低需 8GB，或仅启动核心服务 |
| 前端白屏 | 检查 Nginx 配置，`npm run build` 后需 `docker cp` 到容器 |
| hf-mirror 解析失败 | `.env` 设置 `HF_ENDPOINT=https://huggingface.co` |
