## 二、项目总体技术目标与架构概览

### 2.1 技术目标（技术角度的“一句话”）

> 构建一套以 **胰腺炎单病种**为核心的医学知识图谱与大模型问答系统，实现：
> 
> 1. 自动化从医学文献/指南中抽取实体与关系，构建可查询的知识图谱；  
> 2. 将知识图谱与文献检索能力封装为 **RAG 工具**，并接入 **LangGraph Agent**，支持多轮、可解释的智能问答；  
> 3. 通过 **前后端分离 + Docker 容器化 + 云端部署**，提供可在本地和公网演示的完整系统。

### 2.2 整体架构（后端为主）

整体架构可以粗略分为五层：

- **数据层**：
  - Neo4j：存储知识图谱（节点 = 实体，边 = 关系）；
  - MySQL：存储图谱元信息与快照、用户、历史记录等业务数据；
  - Chroma：存储文档切分后的向量，用于文献/语义检索；
  - 文件系统：上传的 PDF / 文本，以及日志等。

- **知识图谱构建层（KG Builder）**：
  - 从 PDF / 文本中提取正文；
  - 文本预处理 + 分句 + 合段；
  - 基于 LLM 的实体/关系抽取；
  - 将结果写入 Neo4j + MySQL。

- **智能问答层（Agent / RAG）**：
  - LangGraph StateGraph 实现 Agent；
  - 封装“图谱检索”“向量检索/文献检索”等为工具函数；
  - Agent 根据对话状态（messages）决定何时调用哪些工具，调用结果再交由大模型综合生成答案。

- **API 服务层（FastAPI）**：
  - `/api/agent/query` / `/api/agent/query/stream`：智能问答；
  - `/api/graph/*` / `/api/kg/*`：图谱相关接口；
  - `/api/search/*`：实体搜索；
  - `/api/upload` / `/api/kg/build`：文件上传与知识图谱构建；
  - `/api/kb/*`：文档知识库（向量检索）；
  - `/api/history/*`：历史记录；
  - `/api/auth/*`：简单注册/登录。

- **前端展示层（Vue2 + Element UI）**：
  - Chat 页：调用 Agent 接口，展示多轮问答；
  - Graph 页：从 MySQL graph_data / `/api/graph/data` 拉取节点/边，用 ECharts 可视化；
  - Search / Upload / History / Login 等页面封装 API 调用与交互逻辑。

## 三、知识图谱构建流程（从 PDF/文本到 Neo4j + MySQL）

这一部分对应代码主要集中在：

- `backend/app.py` 中的 `build_kg_from_file` + `build_kg_background`；
- `kg/builder.py` 中的 `KnowledgeGraphBuilder`；
- `db/neo4j_client.py` / `backend/mysql_client.py`。

### 3.1 数据入口：文件上传与文件信息缓存

1. **上传接口 `/api/upload`（FastAPI）**  
   - 接收前端上传的 PDF/TXT 文件，生成唯一 `file_id`；  
   - 将文件保存到后端的 `UPLOAD_DIR`（`backend/uploads/`）；  
   - 对 PDF 用 `pdfplumber` 提前做一次正文提取，获得 `pdf_text`；  
   - 在内存字典 `uploaded_files` 中缓存：`{ file_id, filename, path, size, pdf_text, is_pdf }`；  
   - 若 MySQL 可用，则在 `history_records` 中记录一次 `type=upload` 的历史。

2. **前端在“数据上传”页拿到 `file_id` 后，可以调用 `/api/kg/build` 发起构建任务。**

### 3.2 构建任务触发：`/api/kg/build` 与后台线程

1. 前端 POST 请求 `{"file_id": "..."} ` 到 `/api/kg/build`；  
2. 后端生成一个 `task_id`，在内存 `tasks` 字典中写入任务初始状态（processing, progress=0, ...）；  
3. 启动后台线程 `build_kg_background(task_id, file_id, user_id)`：  
   - 任务状态置为“正在提取正文”；  
   - 后续所有构建逻辑在该线程中执行，接口立即返回 `task_id`，前端通过 `/api/kg/build/progress` 轮询进度。

### 3.3 正文提取与文本预处理

在 `build_kg_background` 中：

1. 根据 `file_id` 取出 `uploaded_files[file_id]`；  
2. 调用 `_extract_text_for_ingest`：
   - 若是 PDF：
     - 优先使用上传时缓存的 `pdf_text`；
     - 若为空，再次用 `pdfplumber` 打开文件、逐页 `extract_text()` 拼接；
   - 若是 TXT/JSON：按编码读取或解析；
3. 得到 `text`（纯文本字符串）与 `source_type`，若文本为空则任务失败返回。

### 3.4 LLM 驱动的实体与关系抽取：`KnowledgeGraphBuilder.process_text`

`kg/builder.py` 中的 `KnowledgeGraphBuilder` 是图谱构建核心：

1. **文本预处理与分段**：
   - `TextPreprocessor.clean_text`：清理多余空白与特殊字符；  
   - `split_sentences`：按中英文标点分句；  
   - 将句子按约 1000 字进行聚合，形成多个“段落（paragraphs）”。

2. **循环处理每个段落**：
   - 对每个非空 `paragraph` 调用 `_extract_entities_and_relations(paragraph)`：  
     - 内部使用 `LLMClient` 调用 DeepSeek 模型，配合精心设计的 prompt：  
       - 约定输出 JSON 结构：`{"entities": [...], "relations": [...]}`；  
       - `entities` 包含 `name, type, description` 等；  
       - `relations` 包含 `subject, predicate, object` 等。  
   - 去重逻辑：使用 `seen_entities` / `seen_relations` 集合防止同一实体/三元组重复。

3. **写入 Neo4j**：
   - 对每个实体调用 `_create_entity`：  
     - 根据实体类型映射到 Neo4j 节点标签（如 Disease, Symptom, Treatment 等）；  
     - 设置 `name`、`description` 等属性；  
     - 依赖 `Neo4jClient.execute_write` 执行 Cypher 语句。  
   - 对每条关系调用 `_create_relation`：  
     - 按 `subject` / `object` 寻找对应实体节点；  
     - 创建带有 `predicate` 类型的关系边。

4. **统计结果**：  
   - 返回一个 `result` dict，包含：  
     - `entities_created` / `relations_created`  
     - `paragraphs_processed` / `sentences_processed`  
     - 以及全部实体/关系列表 `entities` / `relations`（供 MySQL 快照使用）。

### 3.5 MySQL 中的图谱元信息与快照：`knowledge_graphs` 表

在 `build_kg_background` 末尾，若 MySQL 可用：

1. 调用 `mysql_client.create_graph(...)`：  
   - 生成 `graph_id`（UUID），写入 `knowledge_graphs(graph_id, graph_name, description, data_source, file_path, user_id)`；  
   - 这里 `graph_name` 通常是源文件名，`data_source` / `file_path` 记录 `file_id`。  

2. 调用 `mysql_client.update_graph_status(...)`：  
   - 更新 `status="completed"`，以及 `entity_count` / `relation_count`。

3. 调用 `mysql_client.update_graph_data(graph_id, {"nodes": [...], "relations": [...]})`：  
   - 由 `result["entities"]` / `result["relations"]` 生成用于前端展示的简化结构：  
     - `nodes`：`{id, name, type, category, description}`；  
     - `relations`：`{id, source, target, relation}`；  
   - `graph_data` 字段存 JSON，前端通过 `/api/graph/data` 或 `/api/kg/{graph_id}/visualize` 直接拉取可视化所需数据。

### 3.6 前端如何使用这些图谱数据

- Graph 页调用 `/api/graph/list` 获取当前用户的图谱列表；  
- 选择某个图谱后，通过 `/api/graph/data?graph_id=...` 拿到 `nodes` / `edges`；  
- 使用 ECharts `graph` 系列绘制知识图谱，并在点击节点时使用 `/api/graph/entity/{id}` 拉取详情。

## 四、智能问答实现：RAG + LangGraph Agent 设计

这一部分是项目的技术核心，也是与普通“知识图谱展示项目”的主要区别所在。

### 4.1 设计目标

1. **问答不是“纯大模型瞎编”，而是基于图谱和文献检索结果生成**；  
2. **支持多轮对话**：可以追问“它怎么治”“这个药有什么副作用”等，需要记住前文；  
3. **可扩展**：将来可以新增工具（例如调用外部 API、风险评估函数），不破坏主流程。

为此，我们采用了“**RAG 作为工具，Agent 作为决策者**”的设计：

- RAG：实现若干检索策略（图谱检索、向量检索、混合检索等）；  
- Agent：基于 LangGraph 的 StateGraph，将 LLM 与工具绑定，由 LLM 决定何时调用哪些工具。  

### 4.2 问答接口与基本数据流

#### 4.2.1 接口设计

- `POST /api/agent/query`：非流式问答，返回完整答案；  
- `POST /api/agent/query/stream`：流式问答，基于 SSE 分段推送答案，前端做打字机效果；  
- 请求体示例：

```json
{
  "question": "胰腺炎有什么常见症状？",
  "session_id": "可选的会话ID",
  "deep_think": false
}
```

返回示例（非流式）：

```json
{
  "question": "胰腺炎有什么常见症状？",
  "answer": "胰腺炎的典型症状包括上腹部持续性疼痛，常向背部放射，伴有恶心、呕吐、发热等……",
  "sources": [
    {"type": "graph", "name": "胰腺炎 — 症状子图"},
    {"type": "doc", "name": "急性胰腺炎诊疗指南（节选）"}
  ],
  "processing_time": 1.23
}
```

#### 4.2.2 数据流简述

1. 前端构造 `messages`（当前问题 + 若干历史轮次，带上 session_id）发送到后端；  
2. 后端调用 Agent：`langgraph_agent.invoke(messages)`；  
3. Agent 在内部根据大模型输出决定：
   - 是否调用图谱检索工具；  
   - 是否调用文献/向量检索工具；  
   - 是否直接生成答案；  
   - 若需要多轮调用工具，则在 tools ↔ agent 之间循环；  
4. 最终将 `answer` 与 `sources` 返回给前端，前端在 Chat 界面展示，并记录到历史中（type=chat/agent）。

### 4.3 LangGraph Agent 的具体结构

#### 4.3.1 状态建模

- 使用 LangGraph 的 `StateGraph`，状态主要为：
  - `messages`: 包含系统提示（system）、用户问题（user）、模型回复（assistant）、工具调用信息（tool_calls）、工具结果（tool）。  
  - 可选的 `context` / `metadata`：如当前用户 ID、会话 ID。

#### 4.3.2 节点设计

- **Agent 节点**：  
  - 实质是一次 LLM 调用，使用 DeepSeek 模型，并通过 `bind_tools` 把若干检索工具暴露给模型；  
  - Prompt 中明确告诉模型：  
    - 可以调用哪些工具（例如：`graph_search`、`doc_search` 等）；  
    - 什么时候应该先检索再回答（如涉及专业知识、需要数据支撑的问题）；  
    - 如何组织回答（先给结论，再引用来源）。

- **Tools 节点**：  
  - 接收来自 Agent 的 `tool_calls` 指令（例如：call graph_search with {"query": "..."}）；  
  - 调用后端实现的具体检索函数（见 4.4）；  
  - 将检索结果封装为 `ToolMessage`，追加到 `messages`，再回到 Agent 节点。

#### 4.3.3 边与控制流

- `START -> agent`：初始状态进入 Agent；  
- `agent -> tools`：当 LLM 输出中包含一个或多个 `tool_calls` 时；  
- `tools -> agent`：工具执行完毕，返回结果，再由 LLM 决定是否继续调用工具或生成最终回答；  
- 若 LLM 不再产生新的 `tool_calls`，则 `agent -> END`，输出最终答案。

这种设计的好处是：

- 不再是固定的「先检索文档 N 段 → 拼接 → 一次生成」，而是由模型**按需调用**检索工具；  
- 可以在一次对话中多次检索不同类型的信息（先图谱、再文献、再回图谱），提高灵活性；  
- 结构上便于将来增加新工具（例如调用指南 API、风险评分函数）。

### 4.4 RAG 工具层实现（图谱 + 文献检索）

Agent 用到的工具本质上是若干 Python 函数，它们各自实现一种“检索策略”，返回结构化结果。

#### 4.4.1 图谱检索（Graph Search）

**目标**：给定用户问题或实体名称，从 Neo4j 中查出相关的实体与关系子图，作为回答依据。

**典型做法**：

- 对问题做简单的实体识别/关键词提取（可以复用 LLM 或正则）；  
- 使用 `Neo4jClient.execute_query` 执行 Cypher，例如：
  - 查找名称包含关键字的疾病节点；  
  - 扩展 1–2 跳邻居，获取“症状”“检查”“治疗”“并发症”等相关节点与边；  
- 将结果转换为统一格式：

```json
{
  "type": "graph",
  "nodes": [...],
  "relations": [...],
  "summary": "根据知识图谱，胰腺炎与以下症状、检查、治疗有关……"
}
```

Agent 在看到这个结果后，可以直接引用 `summary`，也可以把 `nodes/relations` 里的关键信息融入回答。

#### 4.4.2 文献 / 向量检索（Doc Search）

**目标**：从预先构建的文档知识库中，按语义相似度找到与问题最相关的段落。

**实现路径**：

1. 文档入库：  
   - 通过 `/api/kb/documents/ingest` 接口，将 PDF/TXT 文档切分为若干 `chunk`；  
   - 使用 Embedding 模型（`BAAI/bge-large-zh-v1.5`）生成向量；  
   - 将向量和元数据写入 Chroma。  

2. 检索：  
   - 工具函数接收查询文本 `q`；  
   - 调用 Chroma 的 `search(query_text=q, k=TOP_K)`；  
   - 返回若干命中结果，每条包含 `text` + `metadata`（来源文献、页码等）。

3. 工具返回结构：

```json
{
  "type": "doc",
  "chunks": [
    {"text": "...段落1...", "source": "指南A 第x页"},
    {"text": "...段落2...", "source": "论文B 第y节"}
  ]
}
```

Agent 在看到这个结果时，会将 `text` 拼入自身 context，中英文回复时可引用 `source` 作为出处。

#### 4.4.3 混合检索与策略

根据问题类型，可以结合图谱与文献检索：

- 例如：  
  - 若问题偏“结构性”：如“胰腺炎有什么并发症？”——更适合优先图谱检索；  
  - 若问题偏“描述性/细节性”：如“XX 药物的不良反应有哪些临床研究结论？”——适合文献检索；  
  - 若问题涵盖二者：Agent 可能先查图谱确定相关实体，再查文献获取详细解释。

在实现上，这部分策略通过 Prompt 告知 LLM，实际行为交由 LLM 在推理中决定（即哪种工具被触发得更多）。

### 4.5 多轮对话与会话管理

- 请求体中可选 `session_id` 字段；  
- 后端在接到请求时，会根据 `session_id` 查找该会话历史消息，将若干轮问答拼接到 `messages` 中一并送入 Agent；  
- 大模型在看到完整上下文后，就能正确理解“它”“这个病”等指代；  
- 回答结束后，会将本轮问答写回历史（MySQL 或内存），供下一轮使用。

### 4.6 流式输出实现（SSE）

对于 `/api/agent/query/stream`：

- 后端在调用大模型时，开启流式模式（streaming）；  
- 每当模型生成一小段文本，就通过 SSE 通道 `yield` 给前端；  
- 前端监听 SSE 事件，将文本逐字/逐句 append 到聊天窗口，实现“打字机”效果；  
- 这样可以显著降低用户的主观等待时间，提升交互体验，也符合“认知负荷控制”的设计目标。

## 五、部署与运行方式（本地完整版 vs 云端精简版）

### 5.1 本地开发 / 完整版

- 使用 `docker-compose.yml` 启动：后端、Neo4j、MySQL、Redis 等；  
- 本地安装 Python 依赖后运行 `python run.py` 启动 FastAPI；  
- 前端通过 `npm run serve` 跑在开发模式，`VUE_APP_API_BASE_URL` 指向 `http://localhost:5001`；  
- 在该模式下可以完整演示：知识图谱构建、Agent 问答、多轮对话、知识库检索等所有功能。

### 5.2 云端部署 / 精简版

- 使用 `docker-compose.cloud-minimal.yml` 在腾讯云 2C2G 机器上启动：`kg-backend` + `kg-neo4j` + `kg-mysql` + `kg-redis`；  
- 前端通过 `npm run build` 生成静态文件，由云上的 Nginx 托管，并将 `/api/*` 反向代理到 `backend:5001`；  
- 出于资源限制（2G 内存），云端主要展示：登录、图谱展示、实体搜索、历史记录等基础功能；  
- Agent 问答与大规模构建链路推荐在本地高配环境演示，云端若调用可能会出现超时或整体卡顿。

## 六、目前存在的不足与后续改进方向

从工程和研究两个角度，当前项目仍有不少可以改进之处：

1. **工程结构方面**：
   - 初版实现时以“先跑通”为主，部分模块耦合偏重（例如图谱构建、历史记录与 API 层逻辑混杂）；  
   - 后续可考虑进一步拆分服务：将 Agent 问答、图谱构建、知识库服务解耦，形成更清晰的模块边界。  

2. **知识图谱质量与标准化**：
   - 目前实体/关系类型参照多篇文献与部分经验设计，尚未完全对齐某一套权威本体（如 SNOMED CT、UMLS 的局部子集）；  
   - 后续可以尝试将图谱与标准术语体系对齐，增强复用性和可比性。  

3. **问答评估与“认知负荷”验证**：
   - 当前系统的认知负荷优化主要体现在界面与交互设计（图谱可视化、分步展示、历史记录等），尚缺乏系统性的用户实验；  
   - 后续可以设计小样本对照实验，对比“无系统”“只看原始文献”“使用本系统”三种条件下用户的理解效率与主观负担。  

4. **Agent 策略与工具调度**：
   - 目前 Agent 调度依赖大模型在 Prompt 下的“自发策略”，在复杂问题时可能出现工具调用不稳定、检索不足或过度检索的情况；  
   - 可以尝试引入显式的“决策节点”“compact 过往工具结果”等机制，提高问答的一致性和效率。

## 七、小结

从技术实现角度，本项目已经完成：

- 以胰腺炎为代表的单病种医学知识图谱的自动构建与可视化展示；  
- 基于 LangGraph + DeepSeek 的 Agent 问答原型，将知识图谱与文献检索能力封装为可组合的 RAG 工具，实现了多轮、可解释问答；  
- 完整的前后端实现与本地/云端部署方案，具备可复现和可演示性；  
- 一定程度上将“认知负荷”理念体现在界面与交互设计中。

后续工作重点将放在：

1. 对现有系统进行小规模用户评估和性能分析，形成完整的结题报告/论文；  
2. 在工程上进一步清理与重构部分模块，为后续扩展到其他疾病或多病种场景打基础。
