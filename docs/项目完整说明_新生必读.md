## 二、项目是什么（用最简单的话说）

### 2.1 一句话概括

我们做的是一个**面向老龄认知障碍人群的低认知负荷知识辅助系统**：用户可以通过网页**查知识图谱、搜实体、问问题**，系统会基于**知识图谱**和**大模型**给出回答；我们特别关注「**老年认知障碍**」及相关照护与患者教育，并希望通过**结构化、可视化的方式**降低公众理解医学知识的难度（即**降低认知负荷**）。

### 2.2 稍微展开一点

- **知识图谱**：把医学知识整理成「谁和谁有什么关系」的一张「关系网」，例如「轻度认知障碍—常见表现—记忆减退」「某照护措施—用于—日常提醒」。这样既方便计算机检索，也方便人看图理解。
- **大模型与智能问答**：用户用自然语言提问（如「认知障碍老人日常要注意什么？」），系统会先到知识图谱和文献里查相关材料，再交给大模型整理成一段易懂的回答，甚至支持多轮追问。
- **前端界面**：用户通过浏览器打开我们的网页，可以登录、查看图谱、搜索实体、上传自己的资料、查看历史记录、在聊天窗口里提问等。所有你「看得见、点得动」的，都是**前端**在做。
- **后端服务**：网页上的每一个操作（登录、查图谱、问问题等）都会向服务器发请求；服务器负责查数据库、调用大模型、把结果返回给网页。这部分用户看不见，但逻辑和数据处理都在这里，我们称之为**后端**。

所以：**前端 = 用户看到的界面和操作；后端 = 处理逻辑、数据库、大模型调用；知识图谱 = 结构化医学知识；大模型 = 能理解问题并组织答案的 AI。** 四者合在一起，就是「智护银龄·忆路康」这类项目的完整形态。

## 三、项目背景与意义（为什么要做这个）

### 3.1 项目来源

本项目属于**大学生创新创业训练计划**（简称「大创」）中的课题。立项时的核心想法是：公众在获取和理解健康知识时，往往面临**信息杂乱、专业术语多、不知道从哪看起**等问题，也就是**认知负荷**较高。我们希望用**知识工程 + 人工智能**的方法，把复杂的专业健康知识变成**有序、可查、可问**的形式，从而降低公众的认知负荷。

### 3.2 认知负荷是什么（重要概念）

**认知负荷**指的是人在完成某项任务时，大脑需要同时处理的信息量和难度。

- **过高**：信息太多、太散、太专业，人会「看不懂、记不住、不知道下一步该干嘛」，容易放弃或理解错误。
- **适中**：信息被组织得好、有重点、有路径，人就能在有限精力下理解并记住关键内容。

我们的设计目标就是：通过**知识图谱（结构化）+ 智能问答（按需回答）+ 清晰界面（减少无关干扰）**，让用户用更少的「脑力」就能获取到有用的老年认知障碍相关健康与照护知识，也就是**降低认知负荷**。

### 3.3 为什么选「老龄认知障碍」

医学与照护知识范围极大，我们不可能在有限时间内覆盖所有领域。因此项目在实施时把范围**收缩到以老龄认知障碍为代表的认知照护场景**上：

- 这样可以在**一个具体领域**里把「数据 → 图谱 → 问答 → 界面」全链路做深做透；
- 认知障碍有明确的表现、检查、照护要点和患者教育需求，适合做成知识图谱和问答；
- 方法和架构以后可以复用到其他疾病或照护场景，但在结题和演示时，我们以老龄认知障碍为主要例子。

立项名称里可能还有「健康知识生成与组织加工」等表述；实际完成的工作更准确的描述是：**健康知识的「组织」（知识图谱）与「智能服务」（问答、检索）**，富媒体形式的「加工」（如 3D、视频）留作后续扩展。

## 四、专业名词解释（按主题分类）

下面按「主题」列出项目中会遇到的术语，并尽量用通俗语言解释。遇到文档其他地方出现的陌生词，可以回到本节查找。

### 4.1 项目与业务相关

- **知识图谱（Knowledge Graph）**  
  一种用「节点 + 边」表示知识的数据结构。例如「轻度认知障碍」是一个节点，「记忆减退」是一个节点，二者之间可以有一条边「常见表现」。很多个这样的节点和边连在一起，就形成一张「知识网络」，既方便计算机按关系查询，也方便人看图理解。我们项目里的知识图谱主要存医学与照护实体（疾病、症状、药物、检查、照护措施等）以及它们之间的关系。

- **实体（Entity）**  
  知识图谱里的「节点」，通常代表一个具体概念，如某一种疾病、某一种症状、某一种药物。例如「轻度认知障碍」「记忆减退」「某药物」都是实体。

- **关系（Relation）**  
  知识图谱里连接两个实体的「边」，表示二者之间是什么关系。例如「轻度认知障碍 —常见表现— 记忆减退」「某措施 —用于照护— 轻度认知障碍」。

- **认知负荷（Cognitive Load）**  
  见 3.2 节。在项目里我们通过「结构化知识 + 问答 + 界面设计」来降低用户理解医学知识时的认知负荷。

- **老龄认知障碍**  
  本项目重点关注的领域。我们构建的知识图谱、示例问答和演示都围绕老年认知障碍（及与之相关的表现、照护、患者教育等）展开。

### 4.2 技术架构相关

- **前端（Front-end）**  
  用户通过浏览器看到的网页和所有交互（按钮、表单、图表、聊天窗口等）。我们项目的前端用 **Vue.js** 开发，运行在用户的电脑或手机浏览器里，负责把用户的操作发给后端，并把后端返回的数据展示出来。

- **后端（Back-end）**  
  运行在服务器上的程序，负责处理业务逻辑：查数据库、调用大模型、算数据等。用户不直接看到后端，但前端的每一个「请求」都会由后端处理并返回结果。我们项目的后端用 **Python** 和 **FastAPI** 编写。

- **API（Application Programming Interface，应用程序接口）**  
  可以理解为「后端对外提供的一整套标准请求方式」。前端要「登录」「查图谱」「问问题」时，不是直接操作数据库，而是按照约定好的「接口」发请求（例如「POST /api/auth/login」表示登录），后端根据接口类型执行相应逻辑并返回结果。这样前后端可以分开开发、用统一「合同」对接。

- **数据库**  
  用来持久化存储数据的软件。我们项目里用到多种数据库，各有分工：
  - **Neo4j**：图数据库，专门存「节点 + 边」形式的知识图谱数据，适合做「查某实体的关系」「查某类关系」等查询。
  - **MySQL**：关系型数据库，存表格形式的数据，例如用户账号、图谱元信息、历史记录列表等。
  - **Chroma**：向量数据库，存文档的「向量表示」，用于按语义相似度做文献检索（和「知识库」功能相关）。

- **Docker**  
  一种「容器」技术，可以把我们的后端、数据库、前端运行环境等打包成一个个独立的运行单元（容器），在不同电脑上用相同方式启动，避免「在我电脑能跑、在别人电脑跑不起来」的问题。我们项目用 **Docker Compose** 一次性启动多个容器（例如 Neo4j、MySQL、后端服务），方便本地和云上部署。

### 4.3 人工智能与问答相关

- **大模型 / LLM（Large Language Model）**  
  能够理解和生成自然语言的大型人工智能模型。我们项目里通过 **DeepSeek** 等大模型的 API 来「理解用户问题」和「根据检索到的材料组织成答案」。你不需要知道模型内部怎么算，只需要知道：我们给模型「用户问题 + 检索到的文本」，它返回一段连贯的回答。

- **Agent（智能体）**  
  在这里指的是一套「能自己决定下一步做什么」的问答系统：不是固定地「先检索再生成」，而是由一个大模型根据当前问题**决定**要不要查图谱、要不要查文献、查完再生成还是再查一轮。我们使用 **LangGraph** 这个框架来编排「LLM + 工具」的流程，所以常叫「LangGraph Agent」。

- **RAG（Retrieval-Augmented Generation，检索增强生成）**  
  一种常见做法：先根据用户问题去**检索**相关材料（例如从知识图谱、从文献里查），再把「问题 + 检索到的材料」一起交给大模型**生成**答案。这样答案有据可依，而不是纯靠模型「瞎编」。我们项目里 RAG 的能力（图检索、文献检索）被做成 Agent 可以调用的「工具」，由 Agent 决定何时调用、调用哪些。

- **图检索**  
  根据用户问题，在 Neo4j 知识图谱里查相关的实体和关系，把查到的子图或文本片段作为「材料」交给大模型。

- **向量检索 / 文献检索**  
  把文档变成数学上的「向量」，用户问题也变成向量，通过「相似度」在知识库文档里找到最相关的段落，再交给大模型。我们项目里用 **Chroma** 存这些向量，用 **BAAI/bge-large-zh-v1.5** 做中文文本的向量化（嵌入）。

- **流式输出（Streaming）**  
  问答时，答案不是等全部生成完再一次性返回，而是「生成一小段就推一小段」到前端，前端像打字机一样逐字显示。这样用户体感更快，也符合「降低认知负荷」里减少等待焦虑的考虑。

### 4.4 数据处理与任务相关

- **Hadoop**  
  一套用于「分布式」处理海量数据的开源框架。简单理解：当要处理的文件特别多、特别大时，可以把任务拆成很多小任务，在多台机器（或本机多个进程）上同时算，最后把结果汇总。我们项目里用 Hadoop 主要是做「PDF 正文提取」「文本清理」「文本分块」等步骤的分布式执行；在云上或单机演示时，可能不启动 Hadoop，改用单机方式处理。

- **Celery**  
  一个「异步任务队列」系统：把一些耗时的任务（例如「为这个大文件构建知识图谱」）放进队列，由后台的 Worker 进程慢慢执行，前端不用一直等着，可以轮询「任务进度」。这样用户上传并点击「构建」后，可以去做别的事，过一会儿再来看是否完成，避免页面长时间卡住。

- **异步（Asynchronous）**  
  指「发起一个操作后不立刻等结果，先去做别的事，等结果好了再处理」。例如「提交构建任务」是异步的：后端收到请求后马上返回「任务已提交，任务 ID 是 xxx」，前端用这个 ID  periodically 去问「进度如何」，直到完成或失败。

### 4.5 部署与运行相关

- **本地运行**  
  在你自己或实验室的电脑上启动后端、数据库和前端，一般用于开发和完整功能演示（包括大模型问答），因为本机资源相对充足。

- **云上部署 / 云主机**  
  把后端和前端部署到互联网上的某台服务器（例如腾讯云轻量应用服务器），这样别人通过公网 IP 或域名就能访问我们的网页，不需要来你电脑前操作。云主机通常资源有限（例如 2 核 2G 内存），所以我们会用「精简版」只跑必要服务，重功能（如大模型问答）可能不做或会超时。

- **精简版 / 云上演示模式**  
  指在资源较小的云主机上，只启动「后端 + MySQL + Redis + Neo4j」，不启动 Hadoop、Celery 等重组件；前端用 Nginx 提供静态页面，主要展示登录、图谱、搜索、历史等，大模型问答在 2G 机器上可能不稳定或超时。这样既能「让别人在公网打开我们的界面」，又不会把服务器跑垮。

- **Nginx**  
  一种常用的 Web 服务器软件。在云上我们用它做两件事：一是把前端的静态文件（网页、JS、CSS）提供给用户；二是把用户对「/api/...」的请求转发到我们的后端，这样前后端可以共用一个域名和端口，避免跨域等问题。

### 4.6 各数据库在我们项目里分别存什么（小结）

| 数据库 | 存什么 | 为什么用这个 |
|--------|--------|--------------|
| **Neo4j** | 知识图谱的节点和边（实体、关系），以及图上的查询 | 图结构适合「谁和谁有什么关系」「某实体的邻居」这类查询，比表格更直观、查询语言也针对图优化 |
| **MySQL** | 用户账号、图谱元信息（名称、来源、创建时间）、图谱的「快照」数据（graph_data，供前端画图）、历史记录、任务状态等 | 关系型表格适合列表、筛选、分页，且和业务逻辑（用户、图谱归属）结合紧密 |
| **Chroma** | 文档片段的「向量」及原文，用于按语义相似度检索 | 向量检索适合「和这句话意思相近的段落」，是文献检索、知识库检索的基础 |
| **Redis** | 会话、缓存、Celery 任务队列等（若使用） | 读写快，适合临时状态和队列，减轻 MySQL 压力 |

## 五、用户视角：打开系统后能做什么

下面从**用户**的角度，按「打开网页 → 使用各项功能」的顺序说明每一步是什么、看到什么、背后大致发生了什么。不涉及具体代码，只讲「做什么」和「为什么这样设计」。

### 5.1 打开网站与登录

- **用户操作**：在浏览器输入我们的网址（本地一般是 `http://localhost:8080`，云上是 `http://公网IP`），打开首页；若需要上传、历史等能力，会先注册/登录。
- **系统在做什么**：前端把页面展示出来；登录时把账号密码发给后端的「登录接口」，后端在 MySQL 的用户表里核对，通过则返回一个 token（令牌），前端保存下来，之后的请求都会带上这个 token，表示「我是谁」。
- **和认知负荷的关系**：登录一次即可，不需要每次操作都重新输入身份信息，减少重复操作带来的外在认知负荷。

### 5.2 知识图谱页面

- **用户操作**：进入「知识图谱」页，可以选择某一次构建产生的图谱，看到节点和边组成的图，可以缩放、拖拽、点某个节点看详情，也可以搜索实体、导出等。
- **系统在做什么**：前端向后端请求「图谱列表」和「某图谱的节点与边数据」；这些数据来自 MySQL 里保存的图谱元信息和 graph_data（节点、边列表），最初是由「知识图谱构建」流程写入的。前端用 ECharts 等库把节点和边画成可视化图。
- **和认知负荷的关系**：用「图」的方式呈现关系，比大段文字更容易形成心智模型，降低理解医学概念关系的固有认知负荷。

### 5.3 实体搜索

- **用户操作**：在「实体搜索」页输入关键词、选择类型（如疾病、症状），得到实体列表，点某个实体可看详情或「在图中查看」。
- **系统在做什么**：前端把关键词和类型发给后端的「实体搜索」接口；后端优先在 Neo4j 里按标签和属性查，若无则从 MySQL 的图谱节点里查，把匹配的实体列表返回；前端展示为卡片或列表。这样用户不需要事先知道实体全名，通过关键词就能找到相关概念。
- **和认知负荷的关系**：提供明确的「搜索入口」和分类，减少用户「不知道该查什么」的外在认知负荷。

### 5.4 智能问答（聊天）

- **用户操作**：在「智能问答」页输入问题（如「认知障碍老人日常要注意什么？」），系统会先显示「正在检索…」类提示，然后逐字或整段返回回答，有时还会标注「来自图谱」「来自文献」。
- **系统在做什么**：  
  1. 前端把问题发给后端的 Agent 接口（流式或非流式）。  
  2. 后端 Agent（LangGraph）让大模型根据问题决定：要不要查图谱、要不要查文献。  
  3. 若决定查，就调用「图检索」「文献检索」等 RAG 工具，把查到的片段交给大模型。  
  4. 大模型综合问题和材料，生成答案；若是流式接口，就一段段推给前端。  
  5. 前端把答案展示在聊天窗口，并把「来源」展示出来（若有）。  
  多轮对话时，会带上 session_id，后端把历史几轮一起给大模型，从而支持「它怎么治？」这种指代追问。
- **和认知负荷的关系**：用户用自然语言提问即可，不需要学习查询语法；答案由系统从图谱和文献中组织，并标注来源，既降低理解成本，又增加可信度（有益认知负荷用在「理解知识」上）。

### 5.5 数据上传与知识图谱构建

- **用户操作**：在「数据上传」页上传 PDF 或文本文件，选择「生成知识图谱」等操作，等待进度条完成，之后可在图谱页或历史记录中看到这次构建的结果。
- **系统在做什么**：  
  1. 前端把文件通过上传接口发给后端，后端存到磁盘并可能落一条上传记录。  
  2. 用户触发「构建知识图谱」后，后端对文件做：正文提取（若是 PDF）、文本分块、对每一块用大模型做「实体与关系抽取」，得到节点和边。  
  3. 把这些节点和边写入 Neo4j，并把图谱元信息和 graph_data 写入 MySQL，供前端「图谱页」和「图谱列表」使用。  
  单文件构建一般是异步的：前端轮询「构建进度」接口，直到完成或失败。批量构建可能走 Hadoop 或 Celery，逻辑类似，只是多文件并行。
- **和认知负荷的关系**：用户只需「上传 + 点一下构建」，不需要关心文本如何分块、如何抽取实体；复杂流程由系统完成，降低操作和理解的负荷。

### 5.6 历史记录

- **用户操作**：在「历史记录」页按类型（问答、图谱查询、上传、构建等）、状态、时间筛选，查看列表，可点某条查看详情或删除。
- **系统在做什么**：前端请求后端的「历史列表」接口，带筛选参数；后端从 MySQL 的历史表中查出当前用户的记录并分页返回。删除会调用后端的删除接口，后端再删数据库。这样用户不必自己记住「上次问了什么」「上次建了哪张图」，减少工作记忆负担。
- **和认知负荷的关系**：历史记录统一管理，避免用户「记不住之前做过什么」的外在认知负荷。

### 5.7 文档知识库（若已开放）

- **用户操作**：创建或选择知识库，上传文档或对已有文档进行「入库」；在检索框输入问题，系统返回相关文献片段；也可以「基于知识库提问」跳转到智能问答并自动带上问题。
- **系统在做什么**：文档入库时，后端对文本做分块和向量化，存入 Chroma；检索时用问题向量在 Chroma 里做相似度检索，返回相关片段。这些能力也可以作为 Agent 的「文献检索」工具被问答调用。
- **和认知负荷的关系**：知识库把「文献」结构化、可检索，用户通过自然语言就能找到相关段落，而不是在大量 PDF 里自己翻，降低检索阶段的认知负荷。

## 六、系统视角：数据与流程是如何打通的

本节从「系统」的角度，不按页面而按「数据流和业务流程」说明：数据从哪里来、经过哪些步骤、变成什么、最后如何被用户使用。仍不涉及具体代码，只讲**步骤、输入输出和原理**。

### 6.0 从「一份资料」到「能问答」的完整时间线（总览）

下面用一条时间线把「数据进系统 → 变成图谱 → 用户能问」串起来，方便建立整体印象。

1. **资料进入系统**  
   - 用户在前端上传 PDF 或文本，或管理员把批量文件放到指定目录。  
   - 输入：原始文件。输出：文件保存在服务器，并可能有一条「上传记录」进 MySQL。

2. **正文提取**  
   - 若是 PDF，用库把里面的文字提出来（忽略纯图片、页眉页脚等）；若是 txt/json，直接读入。  
   - 输入：PDF/文本文件。输出：一整段纯文本。  
   - 原理：PDF 是二进制格式，需要专门解析才能拿到可处理的文字。

3. **文本分块**  
   - 把长文本按段落或固定长度切成多块，每块长度适中（例如几百到一千多字），方便后续一次送给大模型处理。  
   - 输入：长文本。输出：多个「文本块」。  
   - 原理：大模型有输入长度限制，且小块文本更利于精确抽取「这一段里的」实体和关系。

4. **实体与关系抽取**  
   - 对每一块文本，调用大模型，让它识别：这段里出现了哪些**实体**（疾病、症状、药物、检查等），以及这些实体之间是什么**关系**（如「A 是 B 的常见症状」「C 用于治疗 D」）。  
   - 输入：一个文本块。输出：若干（实体1, 关系, 实体2）三元组。  
   - 原理：大模型擅长从自然语言中做「命名实体识别」和「关系分类」，我们利用这一点把非结构化文字变成结构化知识。

5. **写入 Neo4j**  
   - 把得到的三元组变成图：每个实体是一个节点（若已存在则复用），每条关系是一条边，写入 Neo4j。  
   - 输入：三元组集合。输出：Neo4j 里多了一张可查询的图。  
   - 原理：图数据库专门为「节点 + 边」的存储和查询设计，适合做「某实体的所有关系」「两跳内的邻居」等查询。

6. **写入 MySQL**  
   - 把这次构建的「元信息」（来源文件、创建时间、创建者等）以及整张图的节点和边列表（graph_data）存进 MySQL。  
   - 输入：图谱元信息 + 节点/边列表。输出：MySQL 里多了一条图谱记录。  
   - 原理：前端「按某次构建」拉取图谱时，需要列表（有哪些图）和单图数据（节点边列表），用 MySQL 存便于按用户、按 ID 查询和分页。

7. **用户来问问题**  
   - 用户在前端输入「认知障碍老人日常要注意什么？」并发送。  
   - 前端把问题发给后端的 Agent 接口（可能带 session_id）。

8. **Agent 决策与工具调用**  
   - 后端用 LangGraph 跑 Agent：大模型看到问题后，决定调用「图检索」工具；工具去 Neo4j 查与「认知障碍」「照护」相关的实体和关系，返回一段文本（如「认知障碍老人日常照护包括：规律作息、按时服药、防跌倒…」）。  
   - 大模型拿到这段文本后，可以再查文献（Chroma），也可以直接综合成答案。

9. **生成并返回答案**  
   - 大模型根据「用户问题 + 检索到的材料」生成一段连贯回答；若是流式接口，边生成边推给前端。  
   - 前端把答案显示在聊天窗口，并可选显示「来源：图谱 / 文献」。

整条链路总结：**原始资料 → 正文 → 分块 → 实体关系抽取 → 写入 Neo4j + MySQL → 用户提问 → Agent 查图/文献 → 大模型生成答案 → 前端展示。** 这样「资料」和「问答」就打通了。

### 6.1 知识图谱是怎么来的（从原始数据到图谱）

- **输入**：用户上传的 PDF 或文本文件（我们项目里以老年认知障碍/照护相关文献、指南片段等为主）。
- **步骤概览**：  
  1. **正文提取**：若是 PDF，用 pdfplumber 等库把正文文字提出来；若是 txt/json，直接读入。  
  2. **文本分块**：把长文本切成一块一块（例如按段落或固定长度），避免单次送给大模型的文本过长。  
  3. **实体与关系抽取**：对每一块文本，调用大模型，让它识别其中的「实体」（疾病、症状、药物等）以及「关系」（如 A 是 B 的症状、C 用于治疗 D）。这一步是「知识图谱构建」的核心，把非结构化文字变成结构化的（实体, 关系, 实体）三元组。  
  4. **写入图数据库**：把三元组变成 Neo4j 的节点和边，写入 Neo4j。  
  5. **写入业务库**：把图谱的元信息（例如来源文件、创建时间）以及「节点和边的列表」（graph_data）存到 MySQL，方便前端按「某次构建」拉取整张图的数据并画图。
- **输出**：Neo4j 里多了一张可查询的图；MySQL 里多了一条图谱记录；前端「知识图谱」页可以选这张图并展示。

**原理简述**：大模型擅长从自然语言里识别「谁是什么」「谁和谁有什么关系」，我们利用这一点把医学文献里的知识「抽」出来，再用图结构存起来，供后续检索和问答使用。

### 6.2 智能问答是怎么工作的（从问题到答案）

- **输入**：用户在前端输入的自然语言问题（以及可选的 session_id，用于多轮）。
- **步骤概览**：  
  1. **请求进入后端**：前端把问题（和 session_id）发给后端的 Agent 接口（如 `/api/agent/query` 或 `/api/agent/query/stream`）。  
  2. **Agent 编排**：后端用 LangGraph 跑一个「Agent」流程：当前状态是「用户问题 + 历史消息」；先进入「Agent 节点」，即调用大模型，并告诉它「你可以调用的工具有：图检索、文献检索等」。  
  3. **大模型决策**：大模型根据问题决定要不要调用工具、调用哪个。例如问「认知障碍老人日常要注意什么？」可能先调用「图检索」查 Neo4j 里和认知障碍、照护相关的子图。  
  4. **工具执行**：后端执行图检索（或文献检索），把查到的文本/子图整理成一段「工具返回」，再塞回对话状态。  
  5. **循环或结束**：再次进入 Agent 节点，大模型看到「工具返回」后，可以继续调工具，也可以直接生成最终答案。直到大模型不再调工具、只输出回答为止。  
  6. **返回前端**：把最终答案（以及来源信息）返回；若是流式，则边生成边推送，前端逐字显示。
- **输出**：用户看到一段连贯的回答，以及可选的「来自图谱 / 来自文献」等来源标注。

**原理简述**：RAG = 先检索再生成，避免大模型「瞎编」；Agent = 让大模型自己决定「要不要检索、查什么」，比固定流水线更灵活。我们把 RAG 做成 Agent 的工具，这样既保证答案有据可依，又支持多轮、指代和组合多种检索方式。

### 6.3 多轮对话与 session 是什么原理

- **多轮对话**：用户先问「认知障碍老人日常要注意什么？」，系统回答后，用户再问「它怎么照护？」这里的「它」指的就是上一句的「认知障碍老人」。要让大模型理解「它」，必须把**上一轮（甚至前几轮）的问答**一起送给大模型。
- **session（会话）**：后端用 session_id 区分「哪几次问答属于同一轮对话」。同一 session_id 下，历史消息会按顺序存起来，每次新问题时，把「历史消息 + 新问题」一起交给 Agent，这样大模型就能看到上下文，正确理解「它」「这个病」等指代。
- **实现要点**：前端在第一次提问时可以向后端要一个 session_id（或后端在第一次请求时返回），之后同一对话窗口的后续提问都带上这个 session_id；后端根据 session_id 取出历史消息，拼成 messages 再交给 LangGraph，生成完答案后再把本轮问答写回该 session，供下一轮使用。

### 6.4 流式输出是什么原理

- **非流式**：大模型把整段答案全部生成完后，后端一次性把整段文字返回给前端，前端再整段显示。用户会有一段「空白等待」时间。
- **流式**：大模型每生成一小段（例如几个字或一个子句），后端就立刻把这一段推给前端，前端像「打字机」一样逐字或逐句显示。这样用户更早看到内容，体感更快，也符合「降低认知负荷」里减少等待焦虑的设计。
- **技术方式**：我们通过 **SSE（Server-Sent Events）** 或类似方式，在后端和前端之间建立一条「服务器持续推送」的通道，后端每生成一段就写一段到这条通道，前端监听并追加到页面上。接口上对应的是「流式问答」接口（如 `/api/agent/query/stream`）。

### 6.5 前后端是如何配合的

- **前端**：运行在用户浏览器里，负责展示页面、收集用户操作，并按照 API 约定把「请求」（如登录、查图谱、问问题）通过 HTTP 发给后端。  
- **后端**：运行在服务器上，接收请求，查数据库、调大模型、算结果，再通过 HTTP 把「响应」返回给前端。  
- **约定**：请求的「路径」和「参数」由 API 文档约定好（例如 POST /api/auth/login 表示登录，GET /api/graph/data 表示取图谱数据）。前端按约定发请求，后端按约定解析并执行，这样两边只要遵守同一份「合同」就能协同工作，无需前端关心数据库怎么查、大模型怎么调。

### 6.6 系统整体框图（用文字描述）

可以想象成以下几层（从用户到数据）：

- **用户**：在浏览器里打开我们的网页，点击、输入、查看结果。
- **前端**：运行在用户浏览器里，负责把页面画出来、把用户操作变成「请求」发出去，并把后端返回的数据展示出来（图表、列表、聊天内容等）。
- **网络请求（HTTP）**：前端和后端之间通过「请求—响应」通信，例如「GET /api/graph/list」表示「请给我图谱列表」，「POST /api/agent/query」表示「请根据这个问题用 Agent 生成答案」。
- **后端（API 层 + 业务逻辑层）**：接收请求后，根据接口类型去查数据库、调大模型、算结果，再通过 HTTP 把结果返回给前端。这里包括：认证（登录注册）、图谱列表与数据、实体搜索、Agent 问答、上传与构建、历史记录、知识库等。
- **数据与外部服务**：  
  - **Neo4j**：存知识图谱，供图检索；  
  - **MySQL**：存用户、图谱元信息、graph_data、历史记录等；  
  - **Chroma**：存文档向量，供文献检索；  
  - **大模型 API（如 DeepSeek）**：被后端调用，用于实体抽取、问答生成等；  
  - **Redis**（若用）：缓存、会话、Celery 队列等。

所以整体就是：**用户 → 前端(浏览器) → HTTP → 后端(FastAPI) → Neo4j / MySQL / Chroma / LLM**。前端只和「后端」说话，后端再和各个数据库、大模型打交道；这样职责清晰，也便于以后扩展或换掉某一层。

## 七、部署与运行环境（本地 vs 云上）

### 7.1 本地运行

- **含义**：在你自己的电脑或实验室机器上安装 Python、Node、Docker 等，把后端、数据库、前端都跑起来，用 `http://localhost:8080` 访问前端、`http://localhost:5001` 访问后端 API。
- **适用场景**：开发、调试、完整功能演示（包括大模型问答、批量构建等），因为本机资源一般足够。
- **原理**：Docker Compose 可以一键启动 Neo4j、MySQL、Redis、后端等多个容器；前端用 `npm run serve` 在本地起一个开发服务器，通过配置把 API 请求指到本机 5001 端口。

### 7.2 云上部署

- **含义**：在云厂商（如腾讯云）买一台轻量服务器，在上面用 Docker 跑后端和数据库，用 Nginx 提供前端页面，并把 80（网页）、5001（API）等端口在防火墙放开，这样任何人通过公网 IP 就能访问我们的系统。
- **适用场景**：给导师、评委或他人「远程看界面」、做结题演示，不需要对方到你这台电脑前。
- **为什么有「精简版」**：云主机常见配置是 2 核 2G 内存，若把全栈（Hadoop、Celery、大模型问答等）都跑上，内存和 CPU 会非常紧张，容易出现整机卡死或问答超时。因此我们提供「精简版」配置：只启动后端 + MySQL + Redis + Neo4j，不启动 Hadoop/Celery；前端用 Nginx 托管静态文件。这样云上主要展示「登录、图谱、搜索、历史、上传入口」等，大模型问答建议在本机或高配环境演示。

### 7.3 小结

- **本地**：功能最全，适合开发和完整演示。  
- **云上**：面向公网展示界面和主要业务流程，重功能（问答、大规模构建）受资源限制，可在文档中说明「云上为演示模式，完整问答建议本地演示」。

## 十、小结

- 本项目是一个**面向老龄认知障碍人群的低认知负荷知识辅助系统**（大创主题：智护银龄），以**老年认知障碍**为典型应用，通过**知识图谱 + 大模型 Agent 问答 + 清晰前端**降低公众理解医学与照护知识的**认知负荷**。  
- **前端**负责界面与交互，**后端**负责逻辑与数据，**知识图谱**存结构化医学知识，**大模型与 Agent** 负责理解问题、检索材料并生成答案。  
- 数据流：**原始文献/文本 → 抽取实体关系 → 写入 Neo4j/MySQL → 供图谱展示与检索；用户问题 → Agent 调用图检索/文献检索 → 大模型生成答案 → 前端展示。**  
- 部署上：**本地**可跑全功能，**云上**常用精简版做界面演示，重功能（问答、大规模构建）建议本地或高配环境演示。  
- 结题时把「做了什么、怎么做的、为什么能降低认知负荷」讲清楚，再配上论文/设计书/软著等成果即可；本文档旨在让新成员快速建立整体认知，便于参与后续工作。

如有不清楚的概念或希望展开的章节，可以随时在团队内提出，再对本文档做增补。

## 附录：术语中英对照与一句话解释

| 中文 | 英文 | 一句话解释 |
|------|------|------------|
| 知识图谱 | Knowledge Graph | 用节点和边表示「谁和谁有什么关系」的结构化知识网络 |
| 实体 | Entity | 图谱里的一个节点，代表一个具体概念（如疾病、症状、药物） |
| 关系 | Relation | 图谱里连接两个实体的边，表示二者之间的关系类型 |
| 认知负荷 | Cognitive Load | 人在理解或完成任务时大脑需要处理的信息量与难度 |
| 前端 | Front-end | 用户在浏览器里看到的页面和交互，负责展示和发请求 |
| 后端 | Back-end | 服务器上处理逻辑、查库、调模型的程序 |
| 接口 / API | Application Programming Interface | 前后端约定好的「请求—响应」方式，便于分工协作 |
| 图数据库 | Graph Database | 专门存「节点+边」并能做图查询的数据库（如 Neo4j） |
| 关系型数据库 | Relational Database | 用表格形式存数据的数据库（如 MySQL） |
| 向量数据库 | Vector Database | 存「向量」并按相似度检索的数据库（如 Chroma） |
| 大模型 / LLM | Large Language Model | 能理解和生成自然语言的大型 AI 模型 |
| 智能体 / Agent | Agent | 能根据当前状态「自己决定下一步做什么」的 AI 系统 |
| 检索增强生成 | RAG (Retrieval-Augmented Generation) | 先检索相关材料，再让大模型根据材料生成答案 |
| 流式输出 | Streaming | 答案边生成边推给前端，像打字机一样逐字显示 |
| 会话 | Session | 多轮对话的「同一轮」标识，用于把历史消息拼给大模型 |
| 异步 | Asynchronous | 发起操作后不立刻等结果，先做别的事，结果好了再处理 |
| 容器 | Container (Docker) | 把程序和环境打包成独立运行单元，便于在不同机器上一致运行 |
| 云上演示模式 | - | 在低配云主机上只跑必要服务，主要展示界面，重功能本地演示 |
